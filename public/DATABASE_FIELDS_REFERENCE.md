## 1. profiles - الملفات الشخصية

> جدول يحتوي على معلومات جميع المستخدمين في النظام (مالكين، موظفين، زبائن)

| الحقل | النوع | مطلوب | القيمة الافتراضية | الوصف التفصيلي |
|-------|-------|-------|-------------------|----------------|
| `id` | uuid | ✅ | `gen_random_uuid()` | المعرّف الفريد للسجل - يُولّد تلقائياً ولا يظهر للمستخدم |
| `user_id` | uuid | ✅ | - | معرّف المستخدم في نظام المصادقة (auth.users) - يربط الملف الشخصي بحساب تسجيل الدخول. **لا يوجد foreign key مباشر** لتجنب مشاكل Supabase |
| `name` | text | ✅ | - | **اسم المستخدم الكامل**. يظهر في: رسالة الترحيب في لوحة التحكم (`{displayName} 👋`)، بطاقة الموظف، ملف التقارير PDF |
| `email` | text | ❌ | null | البريد الإلكتروني - يُستخدم داخلياً فقط (النظام يعتمد رقم الهاتف كمعرّف أساسي). يُملأ تلقائياً بصيغة `{phone}@phone.debtflow.local` |
| `phone` | text | ❌ | null | **رقم الهاتف** بصيغة دولية (مثل `+967777123456`). يظهر في: صفحة الإعدادات كحقل مقفل (غير قابل للتعديل)، صفحة تسجيل الدخول |
| `username` | text | ❌ | null | **اسم المستخدم** - يُستخدم فقط للموظفين كمعرّف تسجيل دخول بديل. يظهر في: بطاقة الموظف تحت الاسم، نموذج إنشاء/تعديل الموظف |
| `business_name` | text | ❌ | null | **اسم المنشأة/المحل** - خاص بالمالكين فقط. يظهر في: عنوان لوحة التحكم، رأس التقارير PDF، صفحة تفاصيل المنشأة للزبون، صفحة الإعدادات (قابل للتعديل) |
| `owner_id` | uuid | ❌ | null | **معرّف المالك** - يُستخدم فقط للموظفين لربطهم بمالكهم. عندما يكون `null` = المستخدم ليس موظفاً. يُستخدم في `useEffectiveOwnerId()` لتحديد البيانات التي يراها الموظف |
| `subscription_status` | text | ✅ | `'trial'` | **حالة الاشتراك**. القيم الممكنة: `trial` (فترة تجريبية)، `active` (اشتراك فعال)، `expired` (منتهي). يظهر في: بانر الاشتراك أعلى لوحة التحكم، لوحة الأدمن |
| `is_subscription_active` | boolean | ✅ | `true` | **هل الاشتراك فعال؟** - يُستخدم كعلامة سريعة لمنع الوصول. عندما = `false`: يظهر بانر تحذيري ويُقيّد بعض العمليات |
| `trial_ends_at` | timestamptz | ❌ | null | **تاريخ انتهاء الفترة التجريبية**. يظهر في: بانر الاشتراك (الأيام المتبقية)، لوحة الأدمن (إدارة الفترات التجريبية) |
| `subscription_ends_at` | timestamptz | ❌ | null | **تاريخ انتهاء الاشتراك المدفوع**. يظهر في: بانر الاشتراك، لوحة الأدمن |
| `is_suspended` | boolean | ✅ | `false` | **هل الحساب معلّق؟** - يضبطه الأدمن فقط. عندما = `true`: يُمنع المالك وموظفوه من الوصول كلياً |
| `created_at` | timestamptz | ✅ | `now()` | تاريخ إنشاء الحساب - يظهر في لوحة الأدمن |
| `updated_at` | timestamptz | ✅ | `now()` | آخر تحديث للملف الشخصي |

### أين يُستخدم في الواجهة:
- **صفحة الإعدادات** (`OwnerSettings.tsx`): تعديل `name` و `business_name`، عرض `phone` كحقل مقفل
- **لوحة التحكم** (`OwnerDashboard.tsx`): عرض `business_name` و `name` في رسالة الترحيب
- **التقارير** (`OwnerReports.tsx`): `business_name` في عنوان التقرير PDF
- **لوحة الأدمن** (`AdminOwners.tsx`): جميع حقول الاشتراك والتعليق
- **صفحة الموظفين** (`OwnerEmployees.tsx`): `name`، `username` في بطاقات الموظفين
- **صفحة تفاصيل المنشأة** (`CustomerBusinessDetail.tsx`): `business_name` كعنوان الصفحة

### سياسات الأمان (RLS):
- المستخدم يقرأ ويعدّل ملفه فقط
- المالك يقرأ ملفات موظفيه (`owner_id = auth.uid()`)
- الزبون يقرأ ملفات المالكين المرتبط بهم
- الأدمن يقرأ ويعدّل جميع الملفات
- **لا يمكن حذف** الملفات الشخصية

## 2. user_roles - أدوار المستخدمين

> جدول منفصل للأدوار (لمنع هجمات تصعيد الصلاحيات)

| الحقل | النوع | مطلوب | القيمة الافتراضية | الوصف التفصيلي |
|-------|-------|-------|-------------------|----------------|
| `id` | uuid | ✅ | `gen_random_uuid()` | المعرّف الفريد |
| `user_id` | uuid | ✅ | - | معرّف المستخدم - يرتبط بـ `auth.users.id` |
| `role` | app_role (enum) | ✅ | - | **الدور**. القيم: `super_admin` (مدير النظام)، `owner` (مالك منشأة)، `employee` (موظف)، `customer` (زبون). **يمكن أن يكون للمستخدم أكثر من دور** (مثلاً: owner + customer) |
| `created_at` | timestamptz | ✅ | `now()` | تاريخ إضافة الدور |

### أين يُستخدم في الواجهة:
- **صفحة اختيار الدور** (`Index.tsx`): إذا كان للمستخدم أكثر من دور، يُعرض شاشة اختيار
- **التسجيل** (`Register.tsx`): يُحدد الدور حسب اختيار المستخدم (owner/customer)
- **AuthContext**: يُحمّل الأدوار لتحديد الصفحات المتاحة
- **لوحة الأدمن** (`AdminOwners.tsx`): فلترة المالكين عبر `role = 'owner'`

### القيود:
- **للقراءة فقط** من العميل: لا يمكن INSERT/UPDATE/DELETE من الواجهة
- يُدار فقط عبر Edge Functions (`register-user`, `manage-employee`, `seed-admin`)
- الدالة `has_role()` تُستخدم في جميع سياسات RLS

## 3. customers - الزبائن

> زبائن المنشآت - كل مالك لديه قائمة زبائنه الخاصة

| الحقل | النوع | مطلوب | القيمة الافتراضية | الوصف التفصيلي |
|-------|-------|-------|-------------------|----------------|
| `id` | uuid | ✅ | `gen_random_uuid()` | المعرّف الفريد للزبون |
| `name` | text | ✅ | - | **اسم الزبون**. يظهر في: بطاقة الزبون (الحرف الأول كأفاتار)، القوائم المنسدلة لاختيار الزبون، جداول الديون والمدفوعات، التقارير PDF |
| `phone` | text | ✅ | - | **رقم هاتف الزبون**. يظهر في: بطاقة الزبون مع أيقونة 📞، القوائم المنسدلة. يُستخدم في **الربط التلقائي**: عند إضافة زبون يبحث النظام عن مستخدم مسجّل بنفس الرقم |
| `address` | text | ❌ | null | **عنوان الزبون**. يظهر في: بطاقة الزبون (أسفل البطاقة)، نموذج الإضافة/التعديل |
| `debt_limit` | numeric | ❌ | null | **سقف الدين المسموح**. عندما يكون `null` = لا حد. يظهر في: بطاقة الزبون (`سقف الدين: XXX ر.ي`)، صفحة تفاصيل المنشأة للزبون (تحذير أصفر). **قواعد مهمة**: 1) لا يمكن تعديله إلا بعد مرور 30 يوم على أول حركة مالية (خطأ `DEBT_LIMIT_LOCKED`). 2) يُفحص عند كل دين جديد: إذا (الرصيد الحالي + الدين الجديد > السقف) → يُرفض (`DEBT_LIMIT_EXCEEDED`) |
| `owner_id` | uuid | ✅ | - | **معرّف المالك** - يربط الزبون بمنشأة محددة. كل مالك يرى زبائنه فقط. يُستخدم في جميع استعلامات الفلترة |
| `user_id` | uuid | ❌ | null | **معرّف حساب الزبون** (إذا كان مسجّلاً في التطبيق). عندما يكون `null` = زبون غير مرتبط بحساب. عندما يكون موجود = الزبون يمكنه رؤية ديونه والتفاعل عبر تطبيقه. **الربط التلقائي**: عند إنشاء زبون، تُستدعى `find_customer_user_by_phone()` للبحث عن حساب بنفس الرقم |
| `is_active` | boolean | ✅ | `true` | **حالة التفعيل**. عندما = `false`: يُعرض تحذير ⚠ "غير فعال" في بطاقة الزبون، يختفي من قائمة الزبون في تطبيق الزبائن (لكن بياناته محفوظة عند المالك). يُستخدم بدلاً من الحذف للحفاظ على السجلات |
| `created_at` | timestamptz | ✅ | `now()` | تاريخ إضافة الزبون - يُستخدم في ترتيب القائمة (الأحدث أولاً) |

### أين يُستخدم في الواجهة:
- **صفحة الزبائن** (`OwnerCustomers.tsx`): عرض بطاقات الزبائن مع البحث بالاسم/الرقم، أزرار تعديل/تفعيل/إلغاء
- **نموذج إضافة دين** (`OwnerDebts.tsx`): قائمة منسدلة لاختيار الزبون
- **نموذج تسجيل سداد** (`OwnerPayments.tsx`): قائمة منسدلة تعرض فقط الزبائن الذين عليهم رصيد
- **التقارير** (`OwnerReports.tsx`): فلتر حسب الزبون + تفصيل لكل زبون
- **تطبيق الزبون** (`CustomerBusinesses.tsx`): عرض المنشآت المرتبطة بالزبون

### العمليات المتاحة:
| العملية | الشاشة | الحقول المطلوبة | ملاحظات |
|---------|--------|----------------|---------|
| إضافة | OwnerCustomers | name, phone | address و debt_limit اختيارية |
| تعديل | OwnerCustomers | أي حقل | debt_limit مقفل لمدة 30 يوم بعد أول حركة |
| تفعيل/إلغاء | OwnerCustomers | is_active | بديل عن الحذف |
| حذف | OwnerCustomers | - | يُرفض إذا كان عليه رصيد (`CUSTOMER_HAS_DEBT`). يحذف السجلات المرتبطة بالترتيب: payments → debts → orders → customer |

### الترايجر التلقائي:
- `notify_customer_added`: عند إضافة زبون مرتبط (`user_id != null`)، يُرسل إشعار "تمت إضافتك كزبون في {اسم المنشأة}"

## 4. debts - الديون

> سجلات الديون المسجّلة على الزبائن

| الحقل | النوع | مطلوب | القيمة الافتراضية | الوصف التفصيلي |
|-------|-------|-------|-------------------|----------------|
| `id` | uuid | ✅ | `gen_random_uuid()` | المعرّف الفريد للدين |
| `customer_id` | uuid | ✅ | - | **معرّف الزبون** - foreign key → `customers.id`. يُستخدم في JOIN لعرض اسم الزبون في جدول الديون |
| `owner_id` | uuid | ✅ | - | **معرّف المالك** - يُحدد عبر `useEffectiveOwnerId()` (إذا كان المنشئ موظف، يُستخدم `owner_id` الخاص بالمالك) |
| `amount` | numeric | ✅ | `0` | **مبلغ الدين** بالريال اليمني. يظهر باللون الأحمر في جدول الديون. يُفحص ضد `debt_limit` قبل الحفظ. يُعرض بتنسيق العملة `formatCurrency()` |
| `description` | text | ❌ | null | **وصف الدين** - **إجباري في الواجهة** (يُشترط `description.trim()` قبل الإرسال). يظهر في: عمود الوصف في جدول الديون، تفاصيل الدين في تطبيق الزبون |
| `image_url` | text | ❌ | null | **رابط صورة الدين** - يُخزن في سلة `debt-images`. يظهر كأيقونة 🖼 بجانب الوصف (يفتح في تبويب جديد). يُرفع من الكاميرا أو المعرض عبر `<input type="file" capture="environment">` |
| `created_by` | uuid | ✅ | - | **معرّف المُنشئ** (المالك أو الموظف). يُستخدم في الترايجر لتحديد: إذا كان المُنشئ ≠ المالك → إرسال إشعار للمالك "قام موظفك بتسجيل دين..." |
| `created_at` | timestamptz | ✅ | `now()` | **تاريخ تسجيل الدين**. يظهر في: عمود التاريخ بالجدول، فلتر التاريخ (DateFilter). الترتيب: الأحدث أولاً في العرض، **الأقدم أولاً في توزيع المدفوعات (FIFO)** |

### أين يُستخدم في الواجهة:
- **صفحة الديون** (`OwnerDebts.tsx`): جدول كامل مع فلتر تاريخ، أعمدة: الزبون | المبلغ | الوصف | التاريخ | حذف
- **نموذج إضافة دين**: اختيار الزبون → المبلغ → الوصف (إجباري) → صورة (اختياري)
- **لوحة التحكم** (`OwnerDashboard.tsx`): إجمالي الديون في كارت الإحصائيات
- **تفاصيل المنشأة للزبون** (`CustomerBusinessDetail.tsx`): قائمة الديون مع الصور
- **التقارير** (`OwnerReports.tsx`): تفصيل الديون لكل زبون

### التحققات قبل الإنشاء:
1. **فحص سقف الدين**: `(الرصيد الحالي + المبلغ الجديد) ≤ debt_limit`
2. **فحص الوصف**: `description.trim()` يجب أن يكون غير فارغ
3. **فحص صلاحية الموظف**: `useHasPermission("debts", "add")`

### الترايجرات التلقائية:
- `notify_debt_created`: يُرسل إشعار للزبون "تم تسجيل دين بقيمة XXX ر.ي عليك في {المنشأة}"
- إذا المُنشئ موظف → إشعار للمالك أيضاً

## 5. payments - المدفوعات

> سجلات السداد - كل سداد مرتبط بدين محدد

| الحقل | النوع | مطلوب | القيمة الافتراضية | الوصف التفصيلي |
|-------|-------|-------|-------------------|----------------|
| `id` | uuid | ✅ | `gen_random_uuid()` | المعرّف الفريد |
| `debt_id` | uuid | ✅ | - | **معرّف الدين المُسدّد** - foreign key → `debts.id`. يُحدد تلقائياً بنظام FIFO (الأقدم أولاً). **لا يختاره المستخدم** - النظام يوزع المبلغ تلقائياً |
| `customer_id` | uuid | ✅ | - | **معرّف الزبون** - foreign key → `customers.id`. يُستخدم في JOIN لعرض اسم الزبون |
| `owner_id` | uuid | ✅ | - | **معرّف المالك** |
| `amount` | numeric | ✅ | `0` | **مبلغ السداد**. يظهر باللون الأخضر في جدول المدفوعات. **ملاحظة**: إذا كان المبلغ أكبر من دين واحد، يُقسم على عدة سجلات payments (سجل لكل دين) |
| `description` | text | ❌ | null | **وصف السداد** - اختياري (بخلاف الدين الذي يشترط وصفاً). يظهر في عمود الوصف بالجدول |
| `created_by` | uuid | ✅ | - | **معرّف المُنشئ** |
| `created_at` | timestamptz | ✅ | `now()` | **تاريخ السداد** |

### أين يُستخدم في الواجهة:
- **صفحة المدفوعات** (`OwnerPayments.tsx`): جدول مع فلتر تاريخ
- **نموذج تسجيل سداد**: اختيار الزبون (فقط من لديهم رصيد) → عرض ملخص الحساب → المبلغ → الوصف
- **لوحة التحكم**: إجمالي المدفوعات في كارت الإحصائيات

### آلية التوزيع التلقائي (FIFO):
```
1. جلب جميع ديون الزبون مرتبة بالأقدم أولاً
2. حساب المبلغ المتبقي لكل دين: (مبلغ الدين - مجموع المدفوعات السابقة)
3. توزيع مبلغ السداد:
   - الدين الأول: يُسدد بالكامل أو جزئياً
   - إذا بقي مبلغ → ينتقل للدين التالي
   - يتكرر حتى ينتهي المبلغ
4. كل جزء يُسجل كسطر منفصل في جدول payments
```

### التحققات:
1. **المبلغ > 0**
2. **المبلغ ≤ الرصيد المتبقي** (يُمنع السداد الزائد)
3. **وجود ديون مستحقة** (خطأ `NO_OUTSTANDING_DEBTS`)
4. **صلاحية الموظف**: `useHasPermission("payments", "add")`

### الترايجر:
- `notify_payment_created`: إشعار للزبون "تم تسجيل سداد بقيمة XXX ر.ي"

## 6. orders - الطلبات

> طلبات الزبائن لتسجيل ديون أو مدفوعات - تحتاج موافقة المالك

| الحقل | النوع | مطلوب | القيمة الافتراضية | الوصف التفصيلي |
|-------|-------|-------|-------------------|----------------|
| `id` | uuid | ✅ | `gen_random_uuid()` | المعرّف الفريد |
| `customer_id` | uuid | ✅ | - | **معرّف الزبون المُرسل** - foreign key → `customers.id` |
| `owner_id` | uuid | ✅ | - | **معرّف المالك المُستقبل** |
| `type` | text | ✅ | - | **نوع الطلب**: `debt_request` (طلب تسجيل دين) أو `payment_request` (طلب تسجيل سداد). يُحدد الأيقونة واللون: دين = أحمر 📄، سداد = أخضر 🧾 |
| `amount` | numeric | ✅ | `0` | **المبلغ المطلوب**. يظهر بخط عريض في بطاقة الطلب |
| `description` | text | ❌ | null | **وصف الطلب**. يظهر في خلفية رمادية فاتحة |
| `image_url` | text | ❌ | null | **صورة مرفقة** (إيصال/فاتورة). تُخزن في `debt-images/orders/`. رابط "عرض الصورة" يفتح Lightbox |
| `status` | text | ✅ | `'pending'` | **حالة الطلب**: `pending` (معلّق 🟡)، `approved` (مقبول 🟢)، `rejected` (مرفوض 🔴). يظهر كـ Badge ملون. **فلاتر سريعة** أعلى الصفحة |
| `payment_method_type` | text | ❌ | null | **نوع طريقة الدفع** - لطلبات السداد فقط: `wallet` (محفظة إلكترونية)، `bank` (حساب بنكي)، `cash_point` (نقطة كاش). يظهر في تفاصيل الطلب |
| `payment_method_provider` | text | ❌ | null | **مزوّد الدفع**: `jeeb` (جيب)، `jawali` (جوالي)، `floosak` (فلوسك)، `onecash` (ون كاش)، `cash` (كاش)، `mobimoney` (موبايل موني)، `alkuraimi` (بنك الكريمي). يظهر في تفاصيل الطلب كـ "طريقة الدفع: جيب" |
| `transaction_number` | text | ❌ | null | **رقم العملية/الحوالة** - يُدخله الزبون كدليل سداد. يظهر في تفاصيل الطلب |
| `sender_name` | text | ❌ | null | **اسم المُرسل** - اسم صاحب الحوالة/التحويل. يظهر في تفاصيل الطلب |
| `processed_by` | uuid | ❌ | null | **معرّف المعالج** - من قام بالموافقة/الرفض (المالك أو الموظف). يُملأ تلقائياً عند المعالجة |
| `processed_at` | timestamptz | ❌ | null | **تاريخ المعالجة** |
| `rejection_reason` | text | ❌ | null | **سبب الرفض** - إجباري عند الرفض. يظهر في مربع أحمر مع أيقونة ⚠ |
| `created_at` | timestamptz | ✅ | `now()` | **تاريخ إرسال الطلب** - يُعرض بتنسيق عربي مع الوقت |

### أين يُستخدم في الواجهة:

#### جانب الزبون (`CustomerBusinessDetail.tsx`):
- **زرين أساسيين**: "طلب تسجيل دين" 📄 و "طلب سداد" 🧾
- **نموذج طلب الدين**: المبلغ + الوصف + صورة اختيارية
- **نموذج طلب السداد**: المبلغ + اختيار طريقة الدفع (من طرق المالك) + رقم العملية + اسم المرسل + صورة
- **قائمة الطلبات**: أسفل الأزرار مع حالة كل طلب

#### جانب المالك (`OwnerOrders.tsx`):
- **فلاتر**: معلّق | الكل | مقبول | مرفوض (مع عدّاد المعلّقة)
- **بطاقة الطلب**: النوع + اسم الزبون + التاريخ + المبلغ + الوصف + الصورة + تفاصيل الدفع
- **أزرار الإجراء** (للمعلّقة فقط): ✅ موافقة | ❌ رفض (يطلب سبب)

### دورة حياة الطلب:
```
1. الزبون يُرسل الطلب → status = 'pending'
2. ترايجر notify_new_order يُرسل إشعار للمالك
3. المالك يقرر:
   أ) الموافقة → Edge Function 'process-order':
      - لطلب الدين: يُنشئ سجل في debts
      - لطلب السداد: يُنشئ سجل في payments (مع FIFO)
      - يفحص سقف الدين
      - يُحدّث status = 'approved'
   ب) الرفض → يُحدّث status = 'rejected' + rejection_reason
4. إشعار للزبون بالنتيجة
```
